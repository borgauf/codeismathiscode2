Lambda Calculus Evaluation

Lambda calculus evaluation is the process of simplifying a λ-term by repeatedly applying β-reduction (the only real “computation” rule) until you can’t simplify any further. It is pure substitution — there is no mutable state, no memory cells, no side effects. Each step produces a brand-new term; the old one remains unchanged.

The Basic Rule: β-Reduction

A redex (reducible expression) is any application whose left-hand side is an abstraction:

\[ (\lambda x.M) \, N \]

β-reduction replaces every free occurrence of \(x\) in \(M\) by \(N\):

\[ (\lambda x.M) \, N \quad \to_\beta \quad M[x := N] \]

(We write \(M[x := N]\) for “substitute \(N\) for free \(x\) in \(M\)”.)

𝖟𝕭: Given the redex \((\lambda\ x.x^2)\ 3\), when we replace \(x\) with \(3\) we have reduced the redex to \(3^2\) or \(9\). \(\square\)

Important Safety Step: α-Conversion (renaming)

If \(N\) contains variables that would be accidentally captured by binders inside \(M\), we first rename bound variables in \(M\) (this is always allowed and doesn’t change meaning). This is why we sometimes write \(\lambda x.M \equiv_\alpha \lambda y.M[y/x]\).1 This just means variable names, if replaced consistently, do not change the meaning of a lambda abstraction.

𝖟𝕭: Take the identity function applied to a term:

\[ (\lambda x.x) \, (\lambda y.y) \quad \to_\beta \quad \lambda y.y \quad \square \]

The variable \(x\) is replaced by \(\lambda y.y\); nothing is captured, so we’re done. The result is already in normal form (no more redexes).

Another classic (the K combinator, which discards its second argument):

\[ (\lambda x.\lambda y.x) \, a \, b \quad \to_\beta \quad (\lambda y.a) \, b \quad \to_\beta \quad a \]

Evaluation Strategies

Order matters for efficiency, not for the final answer. If a lambda expression has multiple nested redexes, we can choose which redex to reduce first, second, etc. Two main strategies:

Normal Order (leftmost-outermost, “call-by-name”)

Always reduce the leftmost redex that is not inside another abstraction. This is lazy, i.e., arguments are not evaluated until they are actually needed.

𝖟𝕭: \[ (\lambda x.x) \bigl((\lambda z.z z)(\lambda z.z z)\bigr) \quad \to_\beta \quad (\lambda z.z z)(\lambda z.z z) \]

Here the function argument \(\bigl((\lambda z.z z)(\lambda z.z z)\bigr)\) was not reduced to normal form before it became the argument for \(\lambda x.x\).

Applicative Order

Now we first find the innermost-leftmost redex to start. This is known as call-by-value. Reduce arguments to normal form before substituting. This is what SML, OCaml, Scala, JavaScript, Python, etc. actually do.

𝖟𝕭: (Same as above)

\[ (\lambda x.x) \bigl((\lambda z.z z)(\lambda z.z z)\bigr) \quad \to_\beta \quad \text{(loops forever!)} \]

Both strategies, if they terminate, give the same normal form (Church–Rosser theorem). But normal order can terminate on some programs where applicative order loops forever.

Normal Form and Non-Termination

A term is in normal form when it contains no β-redexes at all, i.e., nothing more to be reduced.

Not every term has one — some reductions never stop.

The famous counter-example (Ω, “omega”):

\[ \omega \equiv \lambda x.x x \qquad \Omega \equiv \omega \, \omega \]

\[ \Omega \quad \to_\beta \quad \Omega \quad \to_\beta \quad \Omega \quad \to_\beta \quad \dots \]

Pure λ-calculus can loop forever, just like any Turing-complete system.

How This Maps to Real Functional Languages (SML Example)

When you type into the SML REPL: 

(fn x => x) (fn y => y);

SML performs applicative-order evaluation (call-by-value) internally, but the mathematical meaning is exactly the β-reduction above. The REPL shows you the final normal form (or a value of the appropriate type). The only difference is that SML also has built-in constants (5, true, +, etc.) and types, but the core reduction engine is still λ-calculus style substitution.

Summary in One Picture

\[

\begin{array}{c} \text{Term} \\ \downarrow \text{(find a redex)} \\ (\lambda x.M) N \\ \downarrow \text{(β-reduce + possible α-renaming)} \\ M[x := N] \quad \leftarrow \text{new term, old one untouched} \\ \downarrow \text{repeat until no redexes left} \\ \text{Normal form (or loop forever)} \end{array}

\]

That single substitution rule, applied over and over, is the entire computational engine of λ-calculus — and therefore of every pure functional language. No mutable state, no “change in place”, just building new terms from old ones by plugging values into functions.

Function Constructions in Lambda Calculus

There are a few more constructions worth naming, and yes — together they are Turing-complete. Here is the full picture.

The Three You Listed

  • Composition: \(f \circ g\), written f o g in SML. Defined by (f o g) x = f (g x). It is actually derived — it can be expressed in terms of abstraction and application alone: \[\lambda f.\lambda g.\lambda x.\ f\ (g\ x)\]
  • Application: The explicit applicator \(\lambda f.\lambda a.\ (f\ a)\). Again, derived from abstraction, but useful as a first-class combinator.
  • Basic Abstraction: \(\lambda x.\ M\), written fn x => ... in SML. This is the primitive construction; the others reduce to it.

Additional Constructions

  • Variables: The atomic case \(x\). Formally a separate syntactic category: every lambda term is a variable, an abstraction, or an application — nothing more.
  • Recursion / Fixed Points: The Y-combinator (or, in a typed setting, SML’s fun / rec keyword) lets a function refer to itself. Bare \(\lambda\)-calculus has no built-in let rec; self-reference must be encoded: \[Y = \lambda f.\ (\lambda x.\ f\ (x\ x))\ (\lambda x.\ f\ (x\ x))\] In SML you write val rec or use fun which desugars to a fixed-point combinator under the hood.
  • Let-binding: let val x = e1 in e2 end in SML is syntactic sugar for an immediate application: \[(\lambda x.\ e_2)\ e_1\] No new power, but essential for readable code.

Do These Enable All Calculations?

Yes. The Church–Turing thesis tells us that untyped \(\lambda\)-calculus is Turing-complete using only abstraction and application. Composition, let, and the Y-combinator are all encodable; they add expressiveness to the programmer but not to the model.

Booleans, natural numbers (Church numerals), pairs, and lists can all be encoded: \[ \mathbf{true} = \lambda t.\lambda f.\ t \qquad \mathbf{false} = \lambda t.\lambda f.\ f \qquad \mathbf{0} = \lambda f.\lambda x.\ x \qquad \mathbf{succ}\ n = \lambda f.\lambda x.\ f\ (n\ f\ x) \]

Beta-Reduction and SML Evaluation

\(\beta\)-reduction is the operational heart of both \(\lambda\)-calculus and SML evaluation. The rule is:

\[(\lambda x.\ M)\ N \;\longrightarrow_\beta\; M[N/x]\]

“Substitute \(N\) for every free occurrence of \(x\) in \(M\).”

Lambda calculus SML equivalent
\(\lambda x.\ M\) fn x => M
\((\lambda x.\ M)\ N\) (fn x => M) N
\(\beta\)-reduction a single evaluation step
normal form a value (no more fn applications pending)
\(\alpha\)-conversion consistent variable renaming (hygienic let)
\(\eta\)-reduction (fn x => f x) \(\equiv\) f (eta-expand/contract)

Evaluation Strategy

  • Pure \(\lambda\)-calculus allows normal-order (outermost-first) or applicative-order (innermost-first) reduction.
  • SML uses call-by-value (applicative order, strict): arguments are reduced to values before substitution. This matches \(\beta\)-reduction with the restriction that \(N\) must already be a value.
  • Haskell uses call-by-need (lazy / normal-order), closer to the theoretical normal form guarantee.

A Concrete Example

(* Beta-reduction made visible *)
(fn x => x + 1) 3
(* ==> (x + 1)[3/x] ==> 3 + 1 ==> 4 *)

val add  = fn x => fn y => x + y   (* curried: λx.λy. x+y *)
val add3 = add 3                    (* partial apply: λy. 3+y *)
val seven = add3 4                  (* beta-reduce: 3+4 = 7  *)

Each function call in SML is a \(\beta\)-reduction step; the SML runtime is, at its core, an engine for performing them in call-by-value order.
