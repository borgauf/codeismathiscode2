\(\Pi\)-type in Agda

In type theory –— whether you are writing proofs in Agda, studying Martin-Löf’s foundations, or looking through the topological lens of Homotopy Type Theory (HoTT) — a \(\Pi\)-type (Pi type) is formally known as a dependent function type.

It is a generalization of the standard function types used in languages like Haskell or Standard ML. In a standard function type \(A \to B\), the type of the output \(B\) is fixed. It does not matter what specific value of \(A\) you pass to the function; the return type is always exactly \(B\).

A \(\Pi\)-type changes this: The type of the output can depend on the value of the input.

Formally, if we have a type \(A\) and a family of types \(B\) that depends on \(A\) (meaning for every value \(x : A\), there is a valid type \(B(x)\)), the \(\Pi\)-type is written as:

\[\Pi_{x : A} B(x)\]

Here is how this single concept manifests across the three specific domains:

In Agda (Programming and Verification)

In Agda, \(\Pi\)-types are the fundamental building blocks of dependent typing. You write them using the syntax (x : A) -> B x or sometimes using the forall keyword.

Consider a function that creates a vector of zeros. The length of the returned vector directly depends on the natural number you pass in as an argument. 

-- Preliminaries
data Bool : Set where
true  : Bool
false : Bool

data Nat : Set where
zero : Nat
suc  : Nat → Nat

-- A standard function (fixed return type)
isEven : Nat -> Bool

-- A dependent function (Pi type)
zeros : (n : Nat) -> Vec Nat n
zeros zero    = []
zeros (suc n) = zero :: zeros n

If you apply zeros to the value 3, the type of the result evaluates strictly to Vec Nat 3. The value 3 has crossed the boundary from the term level into the type level, effectively allowing the compiler to enforce program invariants (like array bounds) statically.

In Martin-Löf Type Theory (Logic)

Under the Curry-Howard correspondence (propositions-as-types), MLTT serves as a foundation for constructive logic. In this framework, \(\Pi\) types correspond exactly to universal quantification (\(\forall\)).

If \(A\) is a domain (like natural numbers) and \(B(x)\) is a proposition about \(x\) (like “\(x\) is even”), then the type \(\Pi_{x : A} B(x)\) represents the logical proposition \(\forall x \in A, B(x)\).

What constitutes a proof of \(\forall x \in A, B(x)\)? It is precisely a dependent function. To prove the statement is true for all \(x\), you must provide a function that takes any element \(a : A\) as an argument and returns a proof of \(B(a)\). If you can write a total, terminating function with this \(\Pi\) type, you have constructively proven the universal theorem.

3. In Homotopy Type Theory (Topology)

HoTT provides a geometric interpretation of Martin-Löf Type Theory, where:

  • Types are spaces.
  • Terms are points in those spaces.
  • Equalities are continuous paths between points.

In this view, a dependent type family \(B(x)\) over a base space \(A\) is interpreted as a fibration (or a fiber bundle). For each point \(x\) in the base space \(A\), there is a “fiber” situated over it, which is the space \(B(x)\).

A \(\Pi\) type \(\Pi_{x : A} B(x)\) is interpreted as the space of continuous sections of this fibration. A section is a mapping that assigns to every point \(x\) in the base space \(A\) a specific point in the fiber above it, \(B(x)\). It is a topological function that “chooses” a point in each dependent space in a continuous manner.

—

Summary

To tie it all together, \(\Pi_{x : A} B(x)\) is a single concept viewed from three angles:

  • Agda: A function (x : A) -> B x where the return type is computed from the input value.
  • MLTT: The universal quantifier \(\forall\), proven by mapping an element to a proof about that specific element.
  • HoTT: The space of cross-sections of a fiber bundle over a base space.

Propositions as types

But if \(\Pi_{x : A} B(x)\) is about types, where \(A\) is a type and \(B\) is a “family of types”, how can \(B\) also be considered a proposition? In the creation of a Vec of three elements, i.e., Vec Nat 3, the 3-tuple vector of Nat elements is a member of the “family of Nat vectors” But above we have the example of:

If \(A\) is a domain (like natural numbers) and \(B(x)\) is a proposition about \(x\) (like “\(x\) is even”), then the type \(\Pi_{x : A} B(x)\) represents the logical proposition \(\forall x \in A, B(x)\).

This has \(B\) as a proposition. This seems to clash with the previous \(A \to B\), type\(\to\)type constructors of a type to a “family of types,” i.e., the mapping of the n to a Vec Nat n member of the “family.” Again, this seems to clash with \(B(x)\) as a proposition in \(\forall x \in A, B(x)\) How can we resolve this?

This is the conceptual leap that makes dependent type theory so powerful and, initially, so mind-bending. This goes to the heart of Curry-Howard Correspondence.

To resolve this clash, we have to change how we define a “proposition.” In Martin-Löf Type Theory, Agda, and HoTT, the foundational rule is: Propositions are Types. There is no distinction between a type (like an integer or a string) and a "logical proposition (like \(x\) is even). They are the exact same kind of mathematical object.

Here is how the concepts of “families of types” and “propositions” perfectly align when viewed through this lens.

1. Propositions as Types—the Even(x) example

In standard classical logic, a proposition is simply evaluated as True or False. But in constructive type theory, a proposition is treated as a Type, and the proofs of that proposition are the Terms (or values) that inhabit that type.1 Exactly what is meant by term is explained below.

The first thing to note is that Even(x) looks like a function. It is a function, a type-level function. It takes a value Nat and evaluates it to a Type, which is usually called Set by Agda.

But because we’re in the land of Propositions as Types we are not defining Even to be some stinking algorithm that merely divides numbers by \(2\) and checks the remainder. No, that would be too gauche; instead, we define Even inductively by laying out the logical rules (axioms) for what makes a number even.

We’ve already established Nat above. Here is the definition of the Even type family 

-- takes a Nat, returns a type...
data Even : Nat → Set where

-- Rule 1: Zero is even.
-- This constructor is a foundational axiom. It provides an
-- immediate valid term (a proof) for the type Even zero.
even-zero : Even zero

-- Rule 2: If n is even, then n + 2 is even.
-- This constructor is an inductive inference rule.
-- It is a function that takes a proof that n is even,
-- and returns a new proof that n + 2 is even.
even-suc2 : {n : Nat} → Even n → Even (suc (suc n))

Let’s look at the proposition \(x\) is even, which we can write as the type Even(x).

  • If \(x = 4\), the type \(Even(4)\) is a valid type. Furthermore, it is an inhabited type. The “values” inside this type are the actual mathematical proofs showing that \(4\) is divisible by \(2\).
  • If \(x = 5\), the type \(Even(5)\) is still a perfectly valid type. However, it is an empty type (analogous to the bottom type \(\bot\)). There are no valid proofs that \(5\) is even, so you can never construct a value of type \(Even(5)\).

Because \(Even(x)\) takes a value \(x\) and returns a Type, \(Even\) is precisely a family of types indexed by the natural numbers.

2. Data as Propositions—the Vec example

Now let’s look at Vec Nat n.

To be precise: Vec Nat is the family of types; it takes a number and returns a type. Vec Nat 3 is a specific Type within that family. A specific array, like 1 :: 2 :: 3 :: [], is a Term (or value) of the type Vec Nat 3.

Under the Curry-Howard Correspondence, the type Vec Nat 3 is also a proposition! It can be read as the logical statement: There exists a sequence of 3 natural numbers.

When you write the code to create the array 1 :: 2 :: 3 :: [], you are writing the proof of that proposition. You proved that a vector of length 3 exists by physically constructing one and handing it to the compiler.

3. Reconciling the \(\Pi\) Type

Now we can bring it back to \(\Pi_{x : A} B(x)\). Because types and propositions are the same thing, \(B(x)\) is both a family of types and a family of propositions.

  • Viewed computationally: \(\Pi_{n : Nat} Vec(Nat, n)\) is the type of a function that takes a number \(n\) and returns an array of length \(n\). (Like the zeros function).
  • Viewed logically: \(\Pi_{x : Nat} Even(x)\) is the type representing the proposition \(\forall x \in Nat, Even(x)\).

To “prove” that all natural numbers are even, you would have to write a function that takes any natural number \(x\) as an input, and returns a valid term of the type \(Even(x)\). Because \(Even(3)\) and \(Even(5)\) are empty types, you will never be able to write a total function that satisfies this \(\Pi\) type. Therefore, the \(\Pi\) type itself remains empty, which means the universal proposition is mathematically unprovable (False).

The beauty of this system is that verifying a mathematical proof and type-checking a program are literally the same algorithm.
