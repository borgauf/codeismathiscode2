Dependent Vector

The Agda type Vec A n (often called a “Dependent Vector” or “Length-indexed List”) is expressible in mathematical set theory.

In set theory, we represent this concept using families of sets and Cartesian products.

What is an indexed family of sets?

The phrase is imported from mathematics, specifically from set theory and category theory, and it has a precise meaning worth unpacking here.

In Mathematics

A family of sets indexed by \(I\) is simply a function \(F : I \to \mathbf{Set}\). That is, for each element \(i \in I\), you get a set \(F(i)\). The word “family” just means “a collection of things organised by an index.” For example:

\[F(n) = \{ \text{lists of integers of length } n \}\]

gives you a family of sets indexed by \(\mathbb{N}\). \(F(0)\) is one set (containing only the empty list), \(F(1)\) is another set (all singleton lists), \(F(2)\) another, and so on. These are genuinely different sets, but they are all related by the same pattern — they are the “same idea” at different indices. The whole collection \(\{ F(n) \mid n \in \mathbb{N} \}\) is the family.

In Agda

Vec A is literally that function. Its type is ℕ → Set, which in Agda means: give me a natural number \(n\), and I will return you a type (a Set). So:

  • Vec A 0 is one type — the type whose only inhabitant is []
  • Vec A 1 is another type — inhabited by things like x ∷ []
  • Vec A 2 is yet another type — inhabited by things like x ∷ y ∷ []

These are distinct types. A term of type Vec A 2 is not a term of type Vec A 3. They are different sets (in the sense of collections of values). But they are all produced by the same type constructor Vec A, applied to different indices. That whole tower of types, organised by \(\mathbb{N}\), is the indexed family.

So the statement “Vec is an indexed family of sets” means precisely:

Vec A is a function \(\mathbb{N} \to \mathtt{Set}\). It does not name a single type; it names a whole collection of types, one per natural number, and those types are organised by — indexed by — their length.

Why “Sets” and not just “Types”?

In Agda’s type theory (Martin-Löf Type Theory), types are sets in a precise sense — Set is Agda’s name for the universe of small types, and it corresponds to the mathematical notion of a set in the constructive and type-theoretic tradition. So “indexed family of sets” and “indexed family of types” mean the same thing here. The set-theoretic language is older and comes from the mathematical foundations; Agda inherits it via the name Set.

The Conceptual Payoff

In Haskell, [a] is a single type and length is a value you can only inspect at runtime. In Agda, there is no single type “list of \(A\)” — there is an entire family of types, one for each possible length, and the type system forces you to always say which member of the family you are working with.

The index \(n\) in Vec A n is not data inside the type; it is a coordinate that selects a particular set out of the family:

\[\mathtt{Vec}\ A : \mathbb{N} \to \mathtt{Set}\]

\(n\) Selected set \(\mathtt{Vec}\ A\ n\)
\(0\) \(\{ \mathtt{[]} \}\)
\(1\) \(\{ x \mathtt{∷} \mathtt{[]} \mid x : A \}\)
\(2\) \(\{ x \mathtt{∷} y \mathtt{∷} \mathtt{[]} \mid x, y : A \}\)
\(\vdots\) \(\vdots\)

Each row is a distinct set. Vec A is the function that produces all of them. That function — the whole tower at once — is the indexed family.

1. The Mathematical Translation

In Agda, Vec is a function from a natural number to a type: Vec : Set → Nat → Set. In Set Theory, for a given set \(A\), the equivalent is a function \(V\) that maps each natural number \(n \in \mathbb{N}\) to a specific set.

The Definition

Let \(A\) be a set. We define the family of sets \(V_n\) (representing Vec A n) by induction:

  1. Base Case (\(n = 0\)): \(V_0 = \{ \emptyset \}\) (The set containing the empty sequence. This corresponds to [].)
  2. Inductive Step (\(n + 1\)): \(V_{n+1} = A \times V_n\) (The Cartesian product of \(A\) and the previous set. This corresponds to _::_.)

Alternatively, in standard set theory, \(V_n\) is simply defined as the set of all functions from the von Neumann ordinal \(n\) to \(A\): \[V_n = A^n = \{ f \mid f: \{0, 1, \dots, n-1\} \to A \}\]

2. Comparison Examples

Let’s look at a vector of natural numbers with a length of 2 containing the elements \(x\) and \(y\).

Agda Example

Give me a type \(A\), and I will give you the set of all $n$-tuples of \(A\)

In Agda, we define the type and the term explicitly. The type system enforces that the length matches the index. “.” 

-- The definition (as provided in your prompt)
data Vec (A : Set) : Nat → Set where
  []   : Vec A zero
  _::_ : {n : Nat} → A → Vec A n → Vec A (succ n)

-- An instance: a vector of length 2
myVec : Vec Nat 2
myVec = 10 :: 20 :: []

When you write myVec : Vec Nat 2, you are telling the Agda compiler: “I am creating a collection of exactly two natural numbers.”

Mathematically, this is equivalent to saying: \[myVec \in \mathbb{N} \times \mathbb{N}\] Or, in the function notation: \[myVec \in \mathbb{N}^{\{0, 1\}}\]

Changing the underlying set (\(A\))

A second example: 

myVec : Vec Float 2
myVec = 10.0 :: 20.0 :: []

By changing Nat to Float, we have changed the codomain of the “function.”

  • In the first case, the “function” maps indices \(\{0, 1\}\) to the set of Natural numbers \(\mathbb{N}\).
  • In the second case, the “function” maps indices \(\{0, 1\}\) to the set of Floating point numbers \(\mathbb{F}\).

The power of Vec is that the length 2 is part of the type. If we tried to do this: 

-- This will cause a COMPILE ERROR
brokenVec : Vec Nat 2
brokenVec = 10 :: []

Agda will look at the definition of [] (which has length zero) and :: (which adds 1 to the length) and realize that 10 :: [] has type Vec Nat 1. Since Vec Nat 1 is not the same type as Vec Nat 2, the code will not compile.

To tie it back to the ZF approach:

  • The Type (Vec Nat 2) is the set of all possible functions from \(\{0, 1\} \to \mathbb{N}\).
  • The Value (10::20::[]) is one specific function in that set: the one where \(f(0) = 10\) and \(f(1) = 20\).
Set Theory Example

In Set Theory (ZF), we usually represent sequences as ordered \(n\)-tuples or functions.

As an \(n\)-tuple: The vector is an element of the \(n\)-fold Cartesian product. \[A^2 = A \times A\] \[(10, 20) \in \mathbb{N}^2\]

As a function (Formal ZF approach): A vector is a function where the domain is the length of the vector. \[f = \{ (0, 10), (1, 20) \}\] Here, \(f \in \mathbb{N}^{\{0, 1\}}\). The “type” Vec Nat 2 is the set of all such functions.

3. Key Differences in Philosophy

While they represent the same idea, the way they function is different:

Feature Agda (Vec A n) Set Theory (\(A^n\))
:--- :--- :---
Nature A Type family. Vec A 3 and Vec A 4 are distinct types. A Set. \(A^3\) and \(A^4\) are distinct sets.
Membership Checked via Type Checking. v :: vs is valid by construction. Checked via Set Membership (\(x \in S\)).
Indexing The length \(n\) is a term inside the type. The length \(n\) is an index for a family of sets.
Failure head [] results in a Compile-time error because the type doesn’t match. head(Ø) is usually undefined or requires a precondition (\(S \neq \emptyset\)).
Summary

To express Vec A n in math:

  1. Define \(\mathbb{N}\) (Natural numbers).
  2. Define \(A^n\) (The set of \(n\)-tuples of \(A\)).
  3. Define the family \(\mathcal{V} = \{A^n\}_{n \in \mathbb{N}}\).

The Agda declaration Vec A n is essentially a constructive, inductive definition of the set of \(n\)-tuples \(A^n\).

Agda Vector vs Haskell List

Haskell List

data List a = Nil | Cons a (List a)

Or using built-in syntax: data [a] = [] | a : [a]

This is a simple inductive type with two constructors. The type carries no information about its length — a [Int] list could be empty or have a million elements; the type alone doesn’t give us any indication.

Agda Vector (length-indexed)

data Vec (A : Set) : ℕ → Set where
  []  : Vec A zero
  _::_ : A → Vec A n → Vec A (suc n)

Key Differences

1. Kind/Sort difference

In Haskell, a list has kind \(* \to *\), taking one type parameter. In Agda, Vec has type \(\mathtt{Set} \to \mathbb{N} \to \mathtt{Set}\), indexed by both a type and a natural number.

In Agda, Vec A n lives in Set, but Vec A has type ℕ → Set — it is a family of types indexed by a natural number. This is a dependent type: the type depends on a value. Haskell’s type system separates the type and value levels, whereas in Agda they live in the same universe and can freely intermix.

2. Length is tracked statically

  • Vec A zero is provably the empty vector
  • Vec A (suc (suc zero)) is provably a 2-element vector
  • A function like head can have type \(\mathtt{Vec}\ A\ (\mathtt{suc}\ n) \to A\), making it total — the empty case is ruled out by the type, so no Maybe wrapper or runtime error is needed

3. Constructor return types differ

In Haskell, both constructors return List a. In Agda:

  • [] returns Vec A zero — it refines the index to \(0\)
  • _∷_ returns Vec A (suc n) — it increments the index by \(1\)

This resembles a generalised algebraic data type (GADT), but in Agda it is the norm rather than a language extension.

4. Dependent pattern matching

When you pattern match on a Vec, Agda learns information about \(n\). Matching on [] tells Agda \(n = 0\); matching on _∷_ tells it \(n = \mathtt{suc}\ k\) for some \(k\). This lets the type checker verify properties such as concatenation preserving length: 

_++_ : Vec A m → Vec A n → Vec A (m + n)
[]       ++ ys = ys
(x ∷ xs) ++ ys = x ∷ (xs ++ ys)

The return type \(\mathtt{Vec}\ A\ (m + n)\) is checked by Agda — you could not accidentally return a vector of the wrong length.

Summary

Property Haskell [a] Agda Vec A n
Kind / Sort \(* \to *\) \(\mathtt{Set} \to \mathbb{N} \to \mathtt{Set}\)
Length in type No Yes, as index \(n\)
head total? No Yes
Dependent types No (without extensions) Yes, natively
Constructor refinement No Yes

The Haskell list is a simple recursive sum type where the length is a runtime value. The Agda Vec is a dependent type where the length is promoted into the type itself, enabling compile-time guarantees about size that are simply inexpressible in vanilla Haskell’s type system.

Dependent types and set theory

Translating dependent types into standard mathematical set theory is the standard way mathematicians build semantic models of type theory.

When you look at Agda’s Vec, you are looking at an indexed family of sets. 

data Vec (A : Set) : Nat → Set where
[]   : Vec A zero
_::_ : {n : Nat} → A → Vec A n → Vec A (succ n)

Here is exactly how Agda’s definition of a vector maps into the language of classical set theory (like Zermelo-Fraenkel set theory).

1. The Type Family:

Vec A : Nat → Set

In Agda, Vec A is a function from a natural number to a Type.

In set theory, let \(A\) be a set, and let \(\mathbb{N}\) be the set of natural numbers. The type family Vec A corresponds to an indexed family of sets. Specifically, it maps each natural number \(n \in \mathbb{N}\) to the \(n\)-fold Cartesian product of \(A\), universally denoted as \(A^n\).

We can define this family recursively in set theory: \[A^0 = \{\emptyset\}\] \[A^{n+1} = A \times A^n\]

So, the Agda signature Vec (A : Set) : Nat → Set simply declares the existence of a mapping: \[V_A : \mathbb{N} \to \mathcal{U}\] (where \(\mathcal{U}\) is the universe of sets), such that \(V_A(n) = A^n\).

2. The Base Constructor:

[] : Vec A zero

In Agda, [] is the sole inhabitant of the type Vec A zero.

In set theory, we look at the set \(A^0\). By definition, \(A^0\) is a set containing exactly one element: the empty sequence (or empty tuple), which is formally represented as the empty set \(\emptyset\).

Therefore, the constructor [] corresponds precisely to that unique element: \[[\;] \iff \emptyset \in A^0\]

3. The Inductive Constructor:

_::_ : {n : Nat} → A → Vec A n → Vec A (succ n)

In Agda, this constructor takes an element, a vector of length \(n\), and builds a vector of length \(n+1\).

In set theory, a function signature with arrows translates to a Cartesian product serving as the domain of a standard function. The implicit {n : Nat} means this is actually a family of functions, one for every \(n \in \mathbb{N}\).

For any given \(n\), the constructor _::_ translates to a function \(cons_n\): \[cons_n : A \times A^n \to A^{n+1}\]

Because \(A^{n+1}\) is defined as \(A \times A^n\), the action of this function is mathematically trivial. It simply takes an element \(x \in A\) and an \(n\)-tuple \(v \in A^n\) and forms an ordered pair: \[cons_n(x, v) = (x, v)\]

The Crucial Distinction: Inductive vs. Extensional

While the translation is exact, the philosophy behind how the two systems treat this construction is different:

  • Set Theory is Extensional: In set theory, \(A^n\) is just a collection of tuples. Set theory doesn’t care how the tuples were built; it only cares about what elements are inside them.
  • Agda (Type Theory) is Intensional/Constructive: Agda defines Vec purely by its constructors (rules for building it). Vec A n is not a pre-existing bucket of tuples; it is the type of all trees that can be built by using [] exactly once and _::_ exactly \(n\) times.

Dependent type theory is often considered a foundational alternative to set theory entirely, leaning heavily into constructive mathematics rather than classical logic.

List vs. Vec in Agda

So if Agda prefers a `Vec A n` data type instead of a Haskell-like list `data List a = [] | Cons List a`, how would one make a Haskell-like list in Agda? I just found it done this way: 

data List ( X : Set) : Set where
  [ ] : List X
  _::_ : X -> List X -> List X

but I see no indexing like with the Vec. In other words, there is no dependent typing. Is this correct?

The definition is the standard “Haskell-style” list. 

data List (X : Set) : Set where
  []   : List X
  _::_ : X -> List X -> List X

1. Is it dependently typed?

Technically, List is a parameterized type, but it is not a “dependent type” in the way Vec is.

  • In List X, the type only cares about what is inside (the X).
  • In Vec X n, the type cares about what is inside AND how many there are (n).

In the List definition, we establish that

2. The “Index” vs. the “Parameter”

Look closely at the first line of each definition:

The List (Non-Dependent)

data List (X : Set) : Set where

  • Everything after the colon (: Set) is the “return type.”
  • It just returns a Set. It doesn’t “carry” any extra information.

The Vec (Dependent)

data Vec (A : Set) : Nat -> Set where

  • Notice the Nat -> Set.
  • This means Vec is a function that takes a Nat and then returns a Set.
  • The Nat is an index. It is baked into the identity of the type.

3. Why use List instead of Vec?

If Vec is “smarter,” why ever use List?

  • Unknown Lengths: If you are reading lines from a file or filtering a collection, you don’t know the length at compile time. List is perfect for this.
  • Simplicity: You don’t have to prove to the compiler that the lengths match up in every function.

5. Summary

In Agda:

  • List Nat is the set of all possible finite sequences of naturals.
  • Vec Nat 2 is only the set of functions from \(\{0, 1\} \to \mathbb{N}\).

The List definition you found is exactly how you do “normal” functional programming in Agda when you don’t need (or want) the compiler to track the exact length of your data.

Polymorphic vs. Monomorphic Vectors

Two versions below are examples of polymorphic vs. monomorphic

1. Version 1: The Polymorphic Vector

data Vec (A : Set) : ℕ → Set where
  []   : Vec A zero
  _::_ : A → Vec A n → Vec A (suc n)
  • Parameter: (A : Set) is a parameter. It is declared once at the top.
  • Scope: Because it is on the left of the colon, A is available inside every constructor ([] and ::).
  • Usage: You can make a Vec Nat 2, a Vec Float 2, or a Vec Bool 2.
  • ZF Analogy: This represents the family of sets \(A^n\) for any set \(A\).

2. Version 2: The Monomorphic Vector

data Vec : ℕ → Set where
  []  : Vec zero
  _∷_ : ∀ {n} → ℕ → Vec n → Vec (suc n)
  • No Parameter: There is no (A : Set) at the top.
  • Hard-coded: Look at the _∷_ constructor. It explicitly takes a ℕ as its first argument.
  • Usage: You can only make a vector of natural numbers. You cannot use this definition to store Float or Bool.
  • ZF Analogy: This represents specifically \(\mathbb{N}^n\).

3. How to “Polymorphic-ize” the second version?

If you wanted the second style but still wanted it to be polymorphic, you would have to move the type A into the constructors as an index.

It would look like this: 

data Vec : Set → ℕ → Set where
  []  : ∀ {A} → Vec A zero
  _∷_ : ∀ {A n} → A → Vec A n → Vec A (suc n)

4. Why the difference? (Parameters vs. Indices)

In Agda, there is a technical distinction:

  • Parameters (like (A : Set) in Version 1): These are “fixed” for the whole structure. They are more efficient for the compiler.
  • Indices (like ℕ in both versions): These can “change” in the constructors (from n to suc n).

5. Summary

  • Version 1 is a Generic Container. It says: “Give me a type \(A\), and I will give you the set of all $n$-tuples of \(A\).”
  • Version 2 is a Specific Mathematical Object. It says: “I am the set of all $n$-tuples of Natural Numbers.”

If you are writing a library, you almost always want Version 1. If you are doing a specific proof about number theory, you might see Version 2.

Agda’s Set and set theory set?

If types in Agda are members of Set, how is Set meant here? Is it in any way like a set theory set?

The short answer is: Conceptually yes, but technically no.

1. The Agda Definition: Set as a “Type of Types”

In Agda, Set is the name for the collection of all “small” types.

  • Nat is a type, so we say Nat : Set.
  • Bool is a type, so we say Bool : Set.
  • List Nat is a type, so we say List Nat : Set.

In Type Theory, we call Set a Universe.

2. Comparison to ZF Set Theory

In ZF Set Theory, a “set” is a collection of elements. In Agda, a “Set” is a collection of types.

Concept ZF Set Theory Agda Type Theory
Basic Unit The Set The Type
Membership \(x \in A\) x : A
Collection of… Elements (numbers, sets) Values (terms)
The “Big” Container There is no “Set of all Sets” Set (The Universe)

3. The Russell’s Paradox Problem

In ZF Set Theory, you cannot have a “Set of all Sets” because it leads to Russell’s Paradox.

Agda avoids this by having a hierarchy of universes:

  • Nat : Set (also called Set0)
  • Set : Set1
  • Set1 : Set2

This prevents the paradox where a set contains itself.

4. Why did Agda name it Set?

The creators of Agda (and Martin-Löf Type Theory) used the name Set to appeal to mathematical intuition.

  • When you write A : Set, you are saying “A is a mathematical set.”
  • When you write x : A, you are saying “x is an element of that set.”

5. The “Function” View Revisited

Remember your textbook’s definition: \[V_n = A^n = \{ f \mid f: \{0, 1, \dots, n-1\} \to A \}\]

In Agda, this translates to:

  • \(A\) is a Set.
  • \(\{0, \dots, n-1\}\) is the type Fin n.
  • The “set of all functions” is the type Fin n -> A.

6. Summary

In Agda, Set is a keyword that means “This thing is a Type.”

When you see data List (X : Set) : Set, read it as: “I am defining a new type called List. It takes a type (X) as input and produces a new type (List X) as output.”

It is “like” set theory because it groups things together, but it is governed by the rules of Logic and Functions rather than the axioms of ZF.
