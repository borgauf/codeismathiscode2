Standard ML 1

Functional programming—theoretically speaking

John Backus’s 1977 Turing Award lecture, titled Can Programming Be Liberated from the von Neumann Style? introduced with great mathematical formalism his proto-functional language FP. Backus was a pioneer who led the team that created Fortran (one of the first major programming languages). After Fortran, Later in his career, he realized that traditional programming, relying heavily on changing variables and managing a computer’s memory state step-by-step, was highly error-prone. He sought to make programming more mathematically elegant. Here are the five basic ingredients he used to encapsulate his FP—with some interpretation for better understanding

  1. A set \(O\) of objects: These are your raw materials, which in programming means data. Objects are the numbers, text strings, or lists that you feed into your program, as well as the results you get out of it.
  2. A set \(F\) of functions: These are your tools or machines. A function’s only job is to take an object (an input) and transform it into a new object (an output). For example, a function that doubles numbers evaluates the object \(5\) and gets the value \(10\).
  3. An application operation: This is the act of actually turning the machinery on. Application just means taking a function and feeding an object into it, i.e., applying the double function to the number \(5\).
  4. A set of functional forms: These are the connectors that let you snap smaller machines together to build bigger ones. Instead of just using one tool, you can hook the output pipe of one function directly into the input pipe of another. (Think of function composition \(f \circ g\).)
  5. A set \(D\) of definitions: This is how you label your new, complex machines. Once you have combined several functions to do something useful, you give it a name so you can reuse it later without having to rebuild it from scratch.

A descriptive breakdown of functional programming

Backus’s definition is foundational, but it is highly theoretical. If you ask a modern software engineer to break down what functional programming actually is, they usually won’t talk about mathematical sets. Instead, they’ll describe this list of core principles:

  • Immutability: Data doesn’t change. Once you create a piece of data, it is locked. If you want to change a list, you do not modify the original list; you create a brand-new copy with the updates. This prevents unexpected bugs where one part of your program accidentally ruins data that another part is trying to use.
  • Pure Functions: A function must follow two strict rules. First, the exact same input will always return the exact same output—just like a math function. Second, functions cannot produce “side effects.” This means functions cannot change things outside of themselves, e.g., modifying a global variable, writing to a database, or printing to the screen.1 This is true of only the purest functional languages. For the sake of usability, there are workarounds.
  • First-Class Functions: Functions are treated just like any other piece of regular data. You can bind a function to a variable (give it a name), pass a function to another function as an argument, or have a function return another function as a result.
  • Declarative Style: You describe to the computer what you want to achieve, rather than have to order it step-by-step what to do. For example, instead of writing a manual loop that says “go through this list one by one, add \(2\) to each number, and save it into a new list,” you simply use a built-in application tool to say, “Apply (map) this addition function to each element of the entire list.”

But then Backus breaks down what objects \(O\) are in more of his math-speak. In Backus’s system, every piece of data (an object, \(O\)) must fall into one of three categories:

  1. atoms,
  2. sequences \(\langle x_1,...,x_n \rangle\) whose elements \(x_i\) are objects,
  3. \(\bot\) (bottom, the undefined object)

  4. Atoms: Think of an atom in chemistry—it is a fundamental, indivisible building block. In functional programming, an atom is a single, standalone value that cannot be broken down any further.2 …just like solving an equation, getting it into its simplest form. The wording above explicitly mentions two example atoms: \(T\) and \(F\), which simply stand for true and false. Other examples of atoms would be basic numbers or characters.
  5. Sequences3 In higher mathematics, a finite sequence can also be viewed as an \(n\)-tuple. Both are ordered lists of elements, such as \(\langle 𝑎_1, a_2,\ldots,a_n \rangle\), where order matters and repeating objects are not considered duplicates. While tuples are typically finite, a sequence is generally considered to be of infinite length \(\langle a_1, a_2, \dots \rangle\) but can be treated as a tuple when limited to a specific length. (\(\langle x_1,\ldots,x_n \rangle\)): A typical example is the humble list, one of the most basic programming data structures. The notation \(\langle x_1,\ldots,x_n \rangle\) is the mathematical way of saying, “A collection of items in a specific order.” Note that the items inside the list (\(x_i\)) are also objects. Because a sequence is an object, and sequences hold objects, this means you can put sequences inside of sequences—over and over recursively—creating lists of lists of lists…4 …which is the actual syntax of the Lisp family of languages,i.e., lists of lists of lists…
  6. \(\boldsymbol{\bot}\): Called bottom,5 The bottom symbol (\(\bot \)), also known as up tack, is a logical constant representing falsity, a contradiction, or the least element in a lattice, commonly used in type theory and computer science to indicate a non-terminating computation, i.e., bombed operation or program. this symbol is the mathematical representation of an error, a crash, or an infinite loop. For example, when a program tries to divide by zero or gets stuck forever, it doesn’t evaluate to an atom or a sequence value; it produces \(\bot\), meaning the result is undefined.

We should note two more specifics about data objects

  • The Empty Sequence (ϕ): The symbol ϕ (phi) represents an empty list (a sequence with zero items inside it). One quirk of this specific system: the empty list is considered both an atom (because “nothingness” is a single, indivisible concept) as well as a sequence. As we get started in SML, we’ll see the empty list [] often.6 In Lisp, this is called nil.
  • Bottom-Preserving Strictness: The proposition, rule, If \(X\) is a sequence containing \(\bot\), then \(X=\bot\) is describing how errors behave in our functional language. Imagine you are packing a box with items. If even one of the items you try to put in the box explodes or causes a fatal error (evaluates to \(\bot\)), the entire box is destroyed. You cannot have a valid sequence or list where even one item is an error, e.g., [1, 2, (⊥), 4]. The moment an error is introduced, the entire sequence instantly becomes an error itself (\(\bot\)). In programming language theory, a system that behaves this way is called strict or “bottom-preserving”.

Now that we have objects \(O\) sorted out—atoms, sequences (e.g., lists), and errors or undefined situations (\(\bot\))—we need tools to actually do something with them.

In Backus’s system, functions are broken down into two main categories: primitive functions, the basic, standalone tools, and functional forms, the connectors used to wire those tools together into an assembly line.

Here is a breakdown of how they work:

  1. Primitive functions are the basic tools. These are the built-in, ready-to-use machines. They take one object in, do a simple task, and spit one object out—and always that same return for that same input guaranteed.

  2. Selectors: In Backus’s mathematical system, numbers are also functions. His function \(1\) means “grab the first item out of a sequence,” i.e., apply the function \(1\) to the sequence \(\langle A, B, C \rangle\), the output is the atom \(A\). In practice these are simply the functions that evaluate a list and return an element based on its order in the list, e.g., SML’s hd, short for head (of list)7 In reality \(1\) as a function name and as an atom, an actual number doesn’t work, hence, programming languages use other names for selectors… 

hd [1,2,3]

val it = 1 : int

or something fancier to return the second element8 This is a pattern-matching anonymous function, all but verbatim from lambda calculus — but it’s not considered “safe” since it doesn’t account for lists of less than two elements. More later… 

(fn (_::x::_) => x) [1,2,3]

val it = 2 : int

In essence, we have an indexing system to automatically identify and retrieve members without the use of loops to march through the sequence/list.

Let’s take a quick in situ math holiday to see how Backus selectors are actually projection functions from higher math

Definition

In mathematics, a projection is a mapping from a set (or algebraic structure) to a subset, which leaves the elements of that subset unchanged. It essentially “squashes” a higher-dimensional object into a lower-dimensional subspace.

Let \( X_1, X_2, \dots, X_n \) be any sets. Their Cartesian product (the set of all n-tuples) is \[ X_1 \times X_2 \times \dots \times X_n = \{ \langle x_1, x_2, \dots, x_n \rangle \mid x_i \in X_i \} \]

A projection function (also written \( \pi_k \) or \( \mathrm{proj}_k \)) is the function that “projects” onto the k-th coordinate:

\[ \pi_k : X_1 \times \dots \times X_n \;\to\; X_k \]

defined by the rule

\[ \pi_k(\langle x_1, x_2, \dots, x_n \rangle) = x_k \]

It throws away every component except the k-th one. That is all there is to it.

  • \( \pi_1 \) always returns the first component.
  • \( \pi_2 \) always returns the second, and so on.

Set Theory The Coordinate Picker

In the context of Cartesian products, a projection function \(pr_i\) (or \(\pi_i\)) selects the $i$-th component of an element.

Given a product \(X \times Y\), the projection \(\pi_1\) is defined as: \[\pi_1: X \times Y \to X, \quad \pi_1(x, y) = x\]

  • Tail (\(tl\)): This chops off the first item of a sequence and returns whatever is left. If you apply \(tl\) to \(\langle A, B, C \rangle\), the output is \(\langle B, C \rangle\).
  • Identity (\(id\)): This is the “do nothing” machine. Whatever you put in, it passes right through unchanged.
  • Predicates (e.g., \(atom\), \(eq\)): These are your inspectors. They look at an object and return either \(T\) (True) or \(F\) (False). For example, the \(atom\) function checks if an object is an atom. If you feed it \(A\), it returns \(T\). If you feed it a sequence like \(\langle A, B \rangle\), it returns \(F\).
  • Functional Forms (The Connectors)

This is where the real magic of functional programming happens. Functional forms are the rules for snapping basic tools together to create complex machines. Backus defined several specific ways to combine them:

  • Composition (\(f \circ g\)): This is hooking two machines up end-to-end like a single conveyor belt. The notation \(f \circ g\) means “run the data through function \(g\) first, and then immediately feed that result into function \(f\).”
  • Construction (\([f_1, f_2, \dots, f_n]\)): This is parallel processing. You take one piece of input data, duplicate it, and send it through several different functions at the exact same time. The results are then automatically packed into a brand-new sequence. For example, if you have a construction made of the functions \([1, tl]\) and you feed it the sequence \(\langle A, B, C \rangle\), it will output a new sequence: \(\langle A, \langle B, C \rangle \rangle\).
  • Apply-to-all (\(\alpha f\)): In modern programming, this is usually called map. It takes a single function (\(f\)) and applies it individually to every single item inside a sequence. If you have a sequence of numbers and you use \(\alpha\) with a “multiply by 2” function, it runs that multiplication on the first number, then the second, then the third, returning a new sequence of doubled numbers.
  • Condition (\(p \rightarrow f ; g\)): This is the functional version of an “if/then” statement. It uses a predicate (\(p\)) to test the input. If the test is True, it routes the data through function \(f\). If the test is False, it routes the data through function \(g\).

Notice what is completely missing from this system: Variables. In traditional programming, you would write code that says “Create a variable X, make it equal to 0, add 1 to X, now save the new X.” In this functional system, there is no storage, no memory states, and no variables changing over time. Data just flows continuously through a pipeline of connected functions, entering as raw input and exiting as a finished result.

Expressions versus commands: the conceptual shift

The imperative model (C/C++): controlling the machine

C and its sister language C++ (C/C++) are modeled heavily on the physical architecture of the computer (specifically, the von Neumann architecture). When you program in C/C++ you are effectively issuing step-by-step commands to directly change the state of the computer’s live memory.

  • Variables as memory boxes: In C/C++, a variable is a named location, a box in memory.
  • Commands as mutations: For example, a line of code such as x = x + 1; is a command.9 In math \(x = x +1\) is an absurdity. Simplifying we would have \(0 = 1\). Crazy. It means: “Go to the memory location named x, read the current value, add 1 to it, and overwrite the contents of that exact same box with the new value.”
int x = 5;
x = x + 1;
// The state of 'x' is now permanently altered to 6
// The previous state of x = 5 is gone forever

The execution of a C/C++ program is a timeline of state changes. To understand what the program is doing at, say, line 50, you have to mentally track the current state of all the mutations10 …all the memory shiftings and overwrites… carried out from lines 1 through 49.

The functional model (ML family): reducing expressions

SML11 Standard ML is the original functional language based on type theory. After SML came Ocaml, Haskell, and F#. is not modeled on hardware architecture, rather, on mathematical logic. When we program in an ML language we do not write commands to change memory; we write expressions to be evaluated, and running an SML program is much like simplifying an algebraic equation.

  • Variables as definitions: In SML, a variable is not a box in memory; it is a mathematical definition. If you write x = 5, you are binding 5 to x. You are stating a truth: “In this scope, x is exactly and forever 5…”
  • Evaluation as Substitution: …then if you have a function f y = y + 1 and you evaluate f x, the program simply substitutes x with 5, evaluating 5 + 1 to 6. No memory locations change state. Consider this ML code
let
  val x = 5
  fun f y = y + 1
in
  print ("x = " ^ Int.toString x ^ "\n" ^ "f x = "
         ^ Int.toString (f x) ^ "\n")
        (* this is just a print statement *)
end

val it = 2 : int

The number 5 did not mutate into, become 6. The original value of x was not destroyed or mutated. Instead, a brand new value (6) was produced as the result of evaluating the function f with the value of x as its argument. This is “pure functional statelessness.” This is immutability. Because values never change once defined, an expression evaluated with the same inputs will always yield the exact same result, with no hidden dependencies on an external, ever-changing memory states.12 Later we’ll learn how “same inputs always produce same outputs” is exactly how mathematical functions behave.

The Compiler’s Magic Trick

If SML programs are just the computer version of mathematical equations, how does the computer actually do the work?

This is where the compiler and the runtime system come in. The SML compiler takes the pure, stateless mathematical logic of the code and translates it into the highly stateful, messy, imperative machine code required by the CPU. But all of this is abstracted away so we don’t have to bother with it.

  • When we bind a variable name with a value or evaluate an expression to a new value, the SML executive runtime allocates new memory behind the scenes—as opposed to C/C++ where all memory management is coded by the programmer.
  • Then when old values are no longer needed, the runtime’s “garbage collector” silently reclaims that memory.

In C, the programmer manages the state. In SML, you work in a pure mathematical domain, and the runtime system handles the dirty work of managing the physical state of the hardware.

In SML, fun f y = y + 1 is nothing more than convenient syntactic sugar for val f = fn y => y + 1. Underneath the hood, all functions in SML are just anonymous lambda functions bound to variable names.

Here is why this design choice is so profound, what its purpose is, and how it forms the direct bridge to mathematical logic.

1. The Purpose: Complete Uniformity

If you define val x = 5, you are binding the integer value 5 to the name x. If you define val f = fn y => y + 1, you are doing the exact same thing: binding a function value (a closure) to the name f.

By making fun just a shorthand for val + fn, the language enforces that functions are not special control structures (as they are in C). They are just values, no different from integers, strings, or lists. Because they are just values, you can naturally pass them as arguments to other functions, return them from functions, or store them in data structures. It makes the entire language beautifully uniform.

2. The Direct Link to Lambda Calculus

You have discovered the exact theoretical foundation of functional programming. Languages like SML, Haskell, and Scheme are not based on the Turing machine model of memory mutation; they are direct, executable implementations of Alonzo Church’s lambda calculus.

In lambda calculus, there are only three fundamental operations:

  1. Variables: \(x, y, z\)
  2. Abstraction (Function creation): \(\lambda y . y + 1\)
  3. Application (Function calling): \((\lambda y . y + 1)\ 5\)

SML’s syntax is a nearly 1-to-1 mapping of this logic:

  • The lambda symbol (\(\lambda\)) becomes the fn keyword.
  • The dot ($.$) becomes the thick arrow (=>), establishing the mapping \(y \mapsto y + 1\).
  • Function application is just writing them next to each other: f x.

3. Computability and Type Theory

This is not just an academic curiosity; it dictates what the computer can actually calculate. In computability theory, the lambda calculus is proven to be Turing-complete. This means that merely by substituting variables into anonymous functions, you can compute anything that a C program can compute with pointers and mutable state.

Furthermore, when you add a rigorous type system on top of this (like SML’s strongly typed integers, lists, and function types), you enter the realm of typed lambda calculi. The expressions you evaluate are no longer just running code; under the Curry-Howard correspondence, they map directly to constructive proofs in first-order logic. Evaluating the expression is equivalent to verifying the proof.

By treating all functions as anonymous lambdas bound to values, SML strips away the hardware-level mechanics of the computer and lets you program directly in the domain of pure mathematical logic.

4. Tossing a Lambda into Fold (Reduce)

Now, let us look at how we toss a lambda into a fold (or “reduce”) function to sum up an entire list of numbers.

In SML, fold comes in two main flavors: foldl (fold left) and foldr (fold right). They both take a function, an initial accumulator value, and a list.

Let’s use foldl to sum the list [1, 2, 3, 4, 5]. We will pass in an anonymous lambda function that takes a tuple (element, accumulator) and adds them together: 

let
  val numbers = [1, 2, 3, 4, 5]
in
  foldl (fn (x, acc) => x + acc) 0 numbers
end

Here is what is happening mathematically:

  1. The foldl function takes our initial accumulator 0 and the first element 1. It applies the lambda: \((1, 0) \mapsto 1 + 0 = 1\).
  2. It takes the new accumulator 1 and the next element 2. Applies the lambda: \((2, 1) \mapsto 2 + 1 = 3\).
  3. It takes the new accumulator 3 and the next element 3. Applies the lambda: \((3, 3) \mapsto 3 + 3 = 6\).

…and so on, until the list is empty, returning the final accumulated value of 15.

If we were to write our own foldl from scratch using pattern matching, it would look like this: 

fun myFoldl f acc [] = acc
  | myFoldl f acc (x::xs) = myFoldl f (f (x, acc)) xs

Notice how our recursive step myFoldl f (f (x, acc)) xs explicitly applies our function f to the current element x and the accumulator acc, and passes that newly evaluated value down as the accumulator for the next recursive call. This completely eliminates the need for stateful, mutating loops!

With SML you do not issue commands. Instead we compose expressions, we evaluate expressions. The difference between composing expressions and issuing commands as with an imperative language is like the difference between a recipe: first do A, then do B, and a mathematical formula: evaluate B and C for a value, then evaluate A with that composed value. A command changes the state of the computer directly, while evaluating expressions is on a layer of abstraction above state changes.

Definition of an expression in SML

⥤ In SML, an expression is a syntactic construct that can be evaluated to produce a value.

This means programs in SML are made up of expressions that are evaluated to give values. What is an expression? Simply put, any section of code that can be evaluated to produce a value.13 In the Lisp family, a form is any part of the code that can, taken by itself, produce a value.

SML expressions are related to mathematical expressions. Consider this math equation

\begin{align*} 7x - 5 = 2 \end{align*}

The two parts on either side of the equals sign, \(7x - 5\) as well as the \(2\), are expressions, while the whole equation itself, \(7x - 5 = 2\), is not typically considered an expression, rather, a formula or an equation. Why? Again, per definition, \(7x - 5\) by itself can be resolved to a value (if we plug a number into \(x\)), while \(2\) is automatically an expression because it resolves to itself.

⥤ To evaluate an expression means to find a value equivalent to the expression.

In math this is reducing to simplest form.14 …also called normal form in higher math. For our \(7x - 5 = 2\) example, by plugging in \(x = 1\) to the expression \(7x - 5\), it evaluates to \(2\). But can’t we then say the equation as a whole evaluates to the Boolean value of true? Yes and no. In just the form of \(7x - 5 = 2\) we have no resolution.

But why is \(2\) an expression? Because anything in the form of a value can be said to evaluate to itself. This is a universal convention of mathematics. If we simply feed possible values—

Composing versus nesting

Another way of contrasting imperative versus functional programming is how code is embedded in code. Imperative programming nests logic while functional composes expressions

  • Nesting Logic: Placing procedure code inside procedure code to control the flow of execution. The focus is on the order of the imperative commands as it changes state.
  • Composing Expressions: Building a result by combining smaller results. The focus is on the result’s definition. Consider this
(* Composing small expressions into a larger one *)
val result = (1 + 2) * (Int.max(5, 10));

Then this value is bound to the variable phrase result However, the entire line val result = (1 + 2) * (Int.max(5, 10)); itself is a declaration that contains expressions. Let’s take a closer look at this distinction. Unlike the formula above, a declaration can never be an expression. Here’s a simple SML declaration

val result = 30 : int
  • Expressions (\(e\)) compute and evaluate to a typed value. They do not, by themselves, introduce new names (e.g., x, y, result) into the global scope.
  • Declarations (\(d\)) modify the environment by binding identifiers (e.g., z, f, myvrbl) to the results of evaluated expressions. Again, they do not evaluate to a value in the same way an expression does; instead, they produce a new environment. Here, x is introduced to the SML system as an immutable variable.

As we see above, the val x = 34; declaration “at the top level”, i.e., in the REPL, “returns” val x = 34 : int as a sort of recipe for making x available.

val x = 34 is a variable binding. Do not think of this as an assignment. In imperative languages like C/C++ an assignment of a number 34 to a variable x establishes x as a name for an internal place in memory, which can subsequently be assigned a different number by the program, overriding the original 34. This is not the case with SML. If later in an SML session we declare val x = 38; we would have literally created a new x that now has the value 38. This means there are now two completely separate x’s. This is called “shadowing”, i.e., some parts of my program might be working with the original x while others with the second x — i.e., this is undesirable.

The code below shows a sequence of bindings 

val x = 34; (* simplest expression is an integer constant *)
val y = 17; (* variable binding *)
val z = (x + y) + (y + 2);
(* dynamic environment: x --> 34, y --> 17 ==> z --> 70 *)
val abs_of_z = if z < 0 then 0 - z else z;

val x = 34 : int

Do not think of these as statements. Imperative languages (C, Java, Python) have statements, while SML has almost none. In imperative languages, an if or a for loop is a statement that does something directly to the state of the program in memory but returns nothing. In SML, expressions return values. For example, with a conditional if ... then ... else

  • statement (Imperative): if (x > 0) { print("hi"); } …if (x > 0) is true this carries out the print and returns nothing,
  • expression (SML): if x > 0 then "hi" else "bye" …returns a string, i.e., a value.

Connection to Abstract Syntax Trees (AST)

Because SML is expression-oriented, the code maps almost 1-to-1 to an AST.

In an AST:

  • Leaves: Constants (34, “hello”) and Variables (x, y).
  • Nodes: Operators or Function calls that take child expressions as sub-trees and produce a new value.

In SML, the “program” is just one giant AST node that evaluates down to a final result. This is why “nesting” is a misleading term — you aren’t nesting instructions; you are building a tree of definitions.

5. The “Everything is an Expression” Rule

In SML, even things that look like blocks are expressions: 

val z = let val x = 10 in x + x end; 
(* The 'let...in...end' block is a single expression evaluating to 20 *)

val x = 34 : int
val y = 17 : int
val z = 70 : int
val abs_of_z = 70 : int
  • Immutability: A binding creates a fixed link between a name (e.g., x) and a value (34). Once bound, the value of that specific x cannot be changed.
  • Shadowing: If you type val x = 50 into the REPL later, you aren’t changing the old x, you are creating a new binding that “shadows” (hides) the previous one in the current scope.
  • Assignment: Actual assignment in SML is only possible using reference cells (e.g., val x = ref 34), where you use the := operator to update the contents of a memory location.

In SML, binding and assignment are distinct concepts. A standard val binding is immutable; you can only “change” it by shadowing the name with a completely new binding. Which is confusing, so don’t do it.

Shadowing (Immutable Bindings)

Shadowing creates a new environment entry rather than updating an existing one. Existing functions continue to use the binding that was in scope when they were defined. For example 

val x = 10;
fun getX () = x;

(* Shadowing x with a new binding *)
val x = 20;

val shadowed_result = x;        (* Evaluates to 20 *)
val function_result = getX (); (* Still evaluates to 10 *)

val z = 20 : int
Reference updates—like with blub languages

Reference cells allow you to update the value stored in a memory location using the := operator. 

val r = ref 10;
fun getR () = !r; (* The ! operator dereferences the cell *)

(* Actual assignment/mutation *)
r := 20;

val assigned_result = !r;      (* Evaluates to 20 *)
val function_result = getR (); (* Now evaluates to 20 *)

How Recursion Replaces Iteration

In functional programming, you do not use for or while loops to change a counter. Instead, you use recursion to pass the “new state” as an argument to the next function call.

1. The Anatomy of a Functional “Loop”

A recursive function replaces the three parts of a traditional loop:

  • Initialization: The initial arguments passed to the function.
  • Condition: The Base Case (e.g., checking for an empty list).
  • Advancement: The recursive call with a modified value (the “step”).
2. Tail Recursion (The “Efficient” Loop)

SML optimized tail-recursive functions to run as efficiently as imperative loops. A function is tail-recursive if the recursive call is the absolute last action it performs. 

(* Imperative-style thinking: sum a list *)
(* In a loop, you'd have: total = 0; for x in list: total += x *)

fun sum_list (nums: int list, acc: int) =
    case nums of
        []      => acc               (* Base Case: Loop is finished *)
      | (x::xs) => sum_list (xs, x + acc) (* Advancement: Next "iteration" *)

val result = sum_list ([1, 2, 3, 4], 0); (* result is 10 *)

3. Why Recursion?
  • No Mutation: You never actually change the value of acc; you just provide a new value for the next function’s acc.
  • Immutability: Each “iteration” has its own unique, immutable scope.
  • Structural Match: Recursion naturally follows the shape of data structures like lists and trees.

In Standard ML (and many other functional languages), the * in a type signature denotes a tuple type, which is the direct programming equivalent of a mathematical Cartesian product.

Here is a breakdown of how the math maps to the code:

  • The Math (\(N \times N\)): The Cartesian product of two sets, representing the set of all ordered pairs \((x, y)\) where both \(x\) and \(y\) are natural numbers.
  • The Code (int * int): The tuple type representing all possible pairs of integers in SML, written as (x, y).

Because of this Cartesian product, a function with the signature int * int -> int actually only takes one argument. That single argument just happens to be a 2-tuple (a pair) containing two integers.17 Technically, SML functions only ever take one argument at a time. It only seems to take multiple arguments by packaging them inside a tuple. Read on about currying…

For example, an addition function with this signature would look like this: 

fun add (x, y) = x + y;
(* val add = fn : int * int -> int *)

When you call add (3, 4), you are passing a single tuple (3, 4) from the Cartesian product int * int into the function.

Uncurried Functions and Cartesian Products

In the uncurried approach, a function takes a single argument that happens to be a tuple.

  • SML Type: int * int -> int
  • Mathematical Relation: A mathematical function is a specific type of binary relation. For an uncurried function taking two arguments, we define a mapping \(f: X \times Y \to Z\).
  • Set Theory Perspective: This function is a subset of the Cartesian product \((X \times Y) \times Z\). Each element of the relation is an ordered pair where the first element is itself an ordered pair \((x, y)\) from \(X \times Y\), and the second is the result \(z\) from \(Z\).
(* Uncurried addition *)
fun add_uncurried (x, y) = x + y;
(* val add_uncurried = fn : int * int -> int *)

val result = add_uncurried (3, 4);
Curried Functions and Higher-Order Mapping

Currying (named after logician Haskell Curry) transforms a function that takes a tuple into a sequence of functions, each taking a single argument. This is the default behavior you see in languages like Haskell, but in SML, you define it explicitly.

  • SML Type: int -> int -> int (which groups right-associatively as int -> (int -> int))
  • Mathematical Relation: Instead of mapping a Cartesian product to a value, a curried function maps a single value to another function. Mathematically, this is \(f: X \to (Y \to Z)\).
  • Set Theory Perspective: If \(Z^Y\) denotes the set of all functions from \(Y\) to \(Z\), our curried function is an element of \((Z^Y)^X\). When you apply the first argument \(x \in X\), the relation yields a new function \(g: Y \to Z\).
(* Curried addition *)
fun add_curried x y = x + y;
(* val add_curried = fn : int -> int -> int *)

(* Partial application is now possible *)
val add_three = add_curried 3;
(* val add_three = fn : int -> int *)

val result = add_three 4;
The Mathematical Isomorphism

The relationship between these two approaches is a fundamental isomorphism in set theory and logic. The set of functions from a Cartesian product is in one-to-one correspondence with the set of functions returning functions:

\[Z^{X \times Y} \cong (Z^Y)^X\]

This means any uncurried function can be mechanically translated into a curried function, and vice versa, without losing any structural information or expressive power.
