* Connecting SML and \(\lambda\)-calculus

** Inner product example

*** What is an Inner Product?

In linear algebra, an inner product is a function that takes two
vectors from a vector space and associates them with a single scalar
quantity (like a real number). Instead of a single formula, an inner
product is defined axiomatically.

For a real vector space, a function denoted as $\langle \vec{u},
\vec{v} \rangle$ is an inner product if it satisfies the following
axioms for all vectors $\vec{u}, \vec{v}, \vec{w}$ and any scalar $c$:

1. *Symmetry:* $\langle \vec{u}, \vec{v} \rangle = \langle \vec{v}, \vec{u} \rangle$
2. *Linearity in the first argument:* - $\langle \vec{u} + \vec{v}, \vec{w} \rangle = \langle \vec{u}, \vec{w} \rangle + \langle \vec{v}, \vec{w} \rangle$
   - $\langle c\vec{u}, \vec{v} \rangle = c\langle \vec{u}, \vec{v} \rangle$
3. *Positive-definiteness:* $\langle \vec{u}, \vec{u} \rangle \geq 0$, and $\langle \vec{u}, \vec{u} \rangle = 0 \iff \vec{u} = \vec{0}$

Any operation that satisfies these strict rules is an inner
product. The standard dot product satisfies these rules, but so do
many other operations (such as integrating the product of two
functions).

*** Example: Inner Product in 3D Space (The Dot Product)

When working with standard vectors in 3D space ($\mathbb{R}^3$), the
standard inner product /is/ the dot product. You calculate it by
multiplying the corresponding components of the vectors and summing
the results.

Let's take two vectors, $\vec{a}$ and $\vec{b}$:
- $\vec{a} = (2, -1, 3)$
- $\vec{b} = (4, 5, -2)$

The standard inner product (dot product) $\langle \vec{a}, \vec{b}
\rangle$ or $\vec{a} \cdot \vec{b}$ is calculated as:

\begin{align*}
\vec{a} \cdot \vec{b} &= (2)(4) + (-1)(5) + (3)(-2) \\
\vec{a} \cdot \vec{b} &= 8 - 5 - 6 \\
\vec{a} \cdot \vec{b} &= -3
\end{align*}

The resulting scalar is *-3*. Geometrically, because the result is negative, it tells us that the angle between these two vectors in 3D space is obtuse (greater than 90 degrees).

*** Example: A Simple Physics Problem

In physics, the concept of *Work* is defined as the inner product (dot product) of the Force vector ($\vec{F}$) and the Displacement vector ($\vec{d}$). 

$$W = \vec{F} \cdot \vec{d}$$

Imagine a child pulling a sled up a 30-degree incline. Let's set up a simple 2D coordinate system where the x-axis runs parallel to the surface of the incline.

1. *Displacement* ($\vec{d}$): The sled moves 10 meters directly up the incline. In our coordinate system, the displacement vector only has an x-component.
   - $\vec{d} = (10, 0)$
2. *Force* ($\vec{F}$): The child pulls on a rope attached to the sled with a force of 20 Newtons. The rope is angled 15 degrees /above/ the incline. To find the vector components of this force, we use trigonometry:
   - $F_x = 20 \cos(15^\circ) \approx 19.32$ N
   - $F_y = 20 \sin(15^\circ) \approx 5.18$ N
   - $\vec{F} = (19.32, 5.18)$

To find the work done by the child on the sled, we calculate the inner product of the two vectors:

$$W = \langle \vec{F}, \vec{d} \rangle = (19.32)(10) + (5.18)(0)$$
$$W = 193.2 + 0$$
$$W = 193.2 \text{ Joules}$$

Only the component of the force that points in the same direction as
the displacement actually contributes to doing work on the sled, which
is exactly the physical property the inner product isolates.

*** Is This an Accurate Generalization of Functional Programming?


- **Composition** :: \( f \circ g \) (in SML: ~f o g~).  Combines two
  functions so that ~(f ∘ g) x = f (g x)~.

- **Application** :: The explicit higher-order applicator \( \lambda
  f.\lambda a.(f\ a) \).  This is the *apply* operator treated as a
  first-class function: it takes a function ~f~ and an argument ~a~
  and returns ~f a~.  For example, ~map~ distributes a function over
  every element of a data structure (e.g., a list).

- **Abstraction** :: Lambda / anonymous functions written as ~fn x =>
  ...~ in SML).

These three correspond directly to the fundamental operations of the
(typed) lambda calculus, which is the theoretical bedrock of ML.

**Examples in Standard ML**

**** Composition

#+BEGIN_SRC sml
val add1 = fn x => x + 1;
val double = fn x => 2 * x;
val doubleThenAdd1 = add1 o double;   (* f ∘ g *)
doubleThenAdd1 5;                     (* → 11 *)
#+END_SRC

**** Application (the applicator λf.λa.(f a))

#+BEGIN_SRC sml
fun apply f a = f a;                  (* explicit λf.λa.(f a) *)
apply double 7;                       (* → 14 *)

(* The "mapping to a list" spirit of generalized application *)
fun map f [] = []
  | map f (x::xs) = (f x) :: map f xs;
map (fn n => n * n) [1, 2, 3];        (* → [1, 4, 9] *)
#+END_SRC

**** Abstraction

#+BEGIN_SRC sml
(fn x => x * x) 5;                    (* → 25 *)
#+END_SRC

**** What else?

While the three operations suffice *in pure theory* to express any
computable function (the lambda calculus is Turing-complete), they do
not represent the full paradigm of functional programming in
ML. Several core features and idioms are missing:

***** Recursion

Pure lambda calculus encodes recursion only via complex fixed-point
combinators (e.g., the Y combinator built from the three
primitives). Real ML provides explicit recursion for practicality.

#+BEGIN_SRC sml
fun factorial 0 = 1
  | factorial n = n * factorial (n - 1);
factorial 5;  (* → 120 *)
#+END_SRC

***** Algebraic Data Types and Pattern Matching

You cannot meaningfully process structured data (lists, trees,
options, etc.) with only composition, application, and
abstraction. ML's expressive power comes from its rich type system and
pattern matching.

#+BEGIN_SRC sml
datatype 'a tree = Empty
                 | Node of 'a * 'a tree * 'a tree;

fun inorder Empty = []
  | inorder (Node(v, left, right)) =
    inorder left @ [v] @ inorder right;
#+END_SRC

***** Idiomatic Higher-Order Library Functions

While ~map~, ~foldl~, ~filter~, etc., can be *derived* from the three
   primitives + recursion, the real day-to-day vocabulary of FP
   consists of these library combinators (not raw λf.λa.(f a)
   applications).

***** Immutability, Purity, Strong Static Typing, and Modules

Referential transparency, Hindley-Milner type inference, and the
   module system (signatures, structures, functors) are central to ML
   but lie outside the three basic operations.

**Conclusion** The three notions — composition, the explicit
application operator \( \lambda f.\lambda a.(f\ a) \), and lambda
abstraction — beautifully summarize the *theoretical core* of how
functions are defined and combined in the lambda calculus. The class
notes likely present them this way to give students a clean, minimal
starting point and to contrast the declarative FP style with
imperative programming.

However, saying functional programming "can be summed up by" only
these three is inaccurate. Practical functional programming in ML (and
similar languages) is about recursive functions over immutable
algebraic data types, extensive use of higher-order library
combinators, strong static typing, and writing programs by composing
meaningful, reusable abstractions.

The lambda-calculus primitives are the engine — but the full ML
paradigm supplies the bodywork that makes the vehicle actually usable
and productive.

*** Do These Three Variations Cover *Every Possible* Function Evaluation in SML?

**Short Answer**
**No — not true.**

They describe the *core mechanisms* for defining and combining pure
functions in the lambda-calculus foundation of SML, but they do
**not** cover every kind of function evaluation that actually occurs
in Standard ML (or any practical lambda-calculus-based language).

**Why the Three Operations Are Not Sufficient**

1. *The Three Are Not Even All Primitive in Pure Lambda Calculus*
   - Pure (typed) lambda calculus only needs *two* syntactic forms:
     - Abstraction: ~λx.M~ (your ~\f~)
     - Application: ~M N~
   - Composition ~f ∘ g~ is *derived*: ~λx. f (g x)~
   - The explicit applicator ~λf.λa.(f a)~ is also *derived* — it is
     just the identity function applied in a curried way.
   - So even in the theoretical core, calling these “three variations”
     that sum up *everything* is a pedagogical simplification, not the
     full formal story.

2. *SML Evaluation Includes Many More Reduction Rules* SML’s
   operational semantics (call-by-value) define function evaluation
   far beyond simple β-reduction of abstractions and
   applications. Examples of evaluation steps *not* covered by the
   three operations:

   ***Recursion (explicit fixed-point)***

   #+BEGIN_SRC sml
fun fact 0 = 1
  | fact n = n * fact (n-1);
fact 5;  (* the recursive call is *not* a plain composition or λf.λa.(f a) *)
   #+END_SRC

   In pure lambda calculus you would need the Y combinator (built from
   the primitives in a very convoluted way). SML treats ~fun~
   recursion as a primitive evaluation rule.

   ***Pattern Matching on Algebraic Data Types***
   #+BEGIN_SRC sml
fun length [] = 0
  | length (x::xs) = 1 + length xs;
   #+END_SRC

   The ~::~ constructor and the ~|~ clauses trigger *case analysis*
   during evaluation — this is not simple function application or
   composition; it is a separate reduction rule in SML’s semantics.

   ***Let-bindings, Sequencing, and Local Definitions***
   #+BEGIN_SRC sml
let val x = 42
    fun f y = x + y
in f 10 end
   #+END_SRC

   The ~let~ binding and the scope resolution are extra evaluation
   steps.

   ***Built-in Primitives and Overloaded Operators***
   
   - ~+~, ~@~ (list append), ~ref~, ~!~, exceptions, I/O, etc.
   - These are not pure λ-applications; they are hard-wired into the
     language runtime.

3. *Higher-Order Library Functions Are Derived, Not Primitive*

   ~map~, ~foldl~, ~filter~ etc. can be *implemented* using
   abstraction + application + recursion, but the actual evaluation of
   ~map f xs~ in the SML compiler/runtime uses optimized,
   non-pure-lambda rules (and pattern matching).

**When the Claim *Would* Be True**

- If we were talking about the *pure untyped lambda calculus* with
  only β-reduction (and perhaps η), then yes — every term is
  ultimately built from abstraction and application, and every
  reduction step is β-reduction.
- The notes are almost certainly using the three as a *teaching
  device* to show students how functions are created, combined, and
  invoked declaratively, before introducing the full ML machinery.

**Conclusion**

The three variations (composition, explicit applicator
~λf.λa.(f a)~, and abstraction) beautifully capture the *declarative
spirit* and the mathematical essence of how you *build* and *invoke*
pure functions. But they do **not** describe “every possible type of
function evaluation” that actually happens when you run an SML
program. Real SML evaluation also involves recursion unfolding,
pattern matching, let-binding, constructor application, and primitive
operations — none of which reduce to just those three forms.

If the notes later introduce these extra constructs (which they almost
certainly do), they are showing how the lambda core is *extended* into
a usable programming language. The three are the engine — but the full
language adds the gearbox, steering wheel, and wheels.  ```
